2. Explain the difference between regression and retesting (5 sentences).
18
19
Retesting is when we test a specific bug after fixing it to make sure it's definitely gone. We run the same tests that we didn't run before. And regression is checking the entire system or part of the functionality to make sure that nothing else is broken after the changes. Retest is about one specific problem, and regression is about overall stability. Both types of testing are usually performed after code changes.

3. Do you think it is possible for a product to conduct only functional testing without checking non-functional requirements?
24
25
Functional testing alone is fine for simple or first versions. Conducting only functional testing is possible, but not always sufficient - it all depends on the type of product, its goals, and users.
Cases when it may be acceptable:
- If it's just a prototype or a test version, you need to quickly test the idea without wasting time on other checks.
- If the product is simple and does not require anything special, such as speed or performance under load.
- If it's a program for internal use in the company and the main task is to simply perform its tasks.
Cases when this may not be enough:
- In critical areas, such as banking or medical systems, even small problems can be serious and lead to serious consequences.
- If you don't check how the system behaves under load, things can go wrong when there are a lot of users.

4. How do you understand the need for smoke testing? Is it always appropriate?
35
36
I understand smoke testing as a quick check of the most important functions of the system to make sure it works at all and is ready for further testing. It's like a preliminary review: whether it starts, whether you can log in, whether key buttons work. Such testing is not always necessary, but in most cases it is very useful. Especially after a new release or an update when you first launch the build, so you don't have to waste time on deep testing if something critical doesn't work right away. It is not always appropriate, for example, when testing very small changes, or if a full regression has already been made and the build is stable. In other words, smoke testing is like a quick check to see if it's worth moving forward at all.

1. In your opinion, what is the greatest benefit of using test management systems? What are their features that allow you to achieve this? Justify your answer in text.
78
79
In my opinion, the biggest benefit of using test management systems is the structured and controlled testing process. They allow not only to perform tests, but also to organize them and track the results and analyze the quality of the product at different stages of development. Thanks to this, teams can not only test but also effectively manage product quality, which, in my opinion, is the main advantage of such systems. Here is an example of features that help to achieve this:

1. Centralized test storage - all tests are stored in one place, which makes it easier to access, update, and reuse them.
2. Test cycles and runs - you can combine tests into logical groups, run them for a specific version of the product, see which tests have passed, failed, or are still pending.
3. Analysis of results - systems provide reports and metrics, which allows you to make more informed decisions.
4. Relationships to bugs and requirements - you can link tests to bugs or user stories, which helps to better understand which parts of the product are covered by tests and which are not.
5. Change history control - you can see who, when, and what changed in tests or results, which is important for process transparency.

1. To Do - The bug has just been logged, not yet assigned to a tester or developer.
2. Open - The bug has received attention and has been assigned to a developer or tester for analysis.
3. In Progress - The bug is being fixed or investigated.
4. Resolved - The bug has been fixed, changes have been made to the code, and is now awaiting testing.
5. Reopened - If the bug re-occurs after being fixed, it returns to the open stage.
6. Closed - The bug has been fixed and tested successfully, and no longer requires attention.

3. Resolved is important to clearly differentiate between fixed bugs and bugs that are waiting to be tested.
4. Reopened allows you to start working again if the bug hasn't been resolved properly.
5. Closed is an important final status that indicates that the issue is completely resolved.
6. Won't Fix is a clear mark indicating a decision not to fix a bug for various reasons.
